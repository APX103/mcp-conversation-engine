# Cognitive Evolution Design: Dreaming & Learning

**Date**: 2026-05-02
**Status**: Approved
**Approach**: Plugin-based Cognitive Modules with Event Bus

## Problem

The current MCP Conversation Engine has a rudimentary memory system:
- Full MEMORY.md stuffed into every system prompt (no selective retrieval)
- Nightly consolidation rewrites all logs via single LLM call (no scoring, no filtering)
- Daily logs deleted after consolidation (all-or-nothing forgetting)
- Skills only created manually or by explicit LLM tool calls (no autonomous evolution)

The agent cannot "dream" (intelligently consolidate memories) or "evolve" (learn from experience).

## Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Architecture | Plugin-based cognitive modules with event bus | Decoupled, independently testable, gradual rollout |
| Storage | Redis + MongoDB + ChromaDB (three-tier) | Working memory + persistence + semantic retrieval |
| LLM for cognitive tasks | Reuse current configured LLM | No new model dependency |
| Autonomy level | Configurable: manual / semi-auto / full-auto | Start semi-auto, graduate to full-auto |
| Phase 1 scope | BM25 retrieval, scoring, shallow sleep + direct promote, skill extraction | Minimum viable learning loop |

## Architecture

```
ConversationEngine (existing, unchanged core)
    │                           │
    │ conversation.end event    │ memory.query request
    ▼                           ▼
CognitiveBus ◄───────── RetrievalEngine
(	event bus / scheduler)      (BM25 + vector hybrid)
    │
    ├─→ MemoryScorer          (importance scoring, 1-5)
    ├─→ DreamEngine            (light sleep → REM → deep sleep)
    │   ├─ DreamCollector      (candidate gathering)
    │   ├─ DreamReflector      (cross-validation with history)
    │   └─ DreamPromoter       (promote to long-term memory)
    └─→ LearnEngine            (experience → reusable skill)
        ├─ LearnExtractor      (method extraction from tool calls)
        ├─ LearnGenerator      (skill document generation)
        └─ LearnIterater       (skill revision on failure)
```

### CognitiveBus Events

| Event | Trigger | Consumers |
|-------|---------|-----------|
| `conversation.end` | Every conversation turn | MemoryScorer, DreamCollector |
| `memory.candidate.scored` | Scoring complete | DreamEngine (threshold check) |
| `dream.rem.start` | Candidates >= threshold | DreamReflector |
| `dream.deep.start` | REM complete | DreamPromoter |
| `learning.extract` | Conversation with 3+ tool calls | LearnExtractor |
| `learning.skill.generated` | Skill document ready | LearnIterater (dedup check) |
| `memory.query` | System prompt construction | RetrievalEngine |

## Storage Layer

### Redis (Working Memory)

```
memory:working:{userId}       # Current session temp memory (list, TTL 24h)
memory:candidate:{userId}     # Candidate memory queue (list, TTL 7d)
dream:stage:{userId}          # Current dream stage (string: idle/light/rem/deep)
retrieval:cache:{hash}        # Retrieval result cache (string JSON, TTL 1h)
```

### MongoDB (Persistent Storage)

New collections:

```javascript
// cognitive_candidates - candidate memories (dream intermediate state)
{
  userId: string,
  content: string,
  score: number,            // 1-5 importance score
  type: 'preference' | 'fact' | 'method' | 'emotion',
  confidence: number,       // 0.0-1.0
  source: string,           // conversation reference
  stage: 'candidate' | 'rem' | 'promoted' | 'discarded',
  createdAt: Date,
  expiresAt: Date
}

// cognitive_skills - auto-generated skills (separate from manual skills)
{
  userId: string,
  name: string,
  description: string,
  content: string,          // full markdown skill document
  version: number,
  confidence: number,       // 0.0-1.0, boosted on successful use
  autoGenerated: boolean,
  sourceConversationIds: string[],
  confirmedAt: Date | null,  // null = pending confirmation
  active: boolean,
  createdAt: Date,
  updatedAt: Date
}
```

Existing collections unchanged: sessions, memories, dailyLogs, commitments, skills.

### ChromaDB (Semantic Memory)

```
Collection: memories
  - id: memory document ID
  - embedding: content vectorization
  - metadata: { userId, type, createdAt, importance }
```

## Dream Engine

### Stage 1: Light Sleep (Collector)

**Trigger**: `conversation.end` event

1. Extract key information fragments from the conversation (user preferences, decisions, knowledge points, emotional tendencies)
2. LLM scores each fragment (1-5, dimensions: importance x persistence x reusability)
3. Fragments scoring >= 3 are written to Redis candidate queue and MongoDB `cognitive_candidates`
4. Each candidate includes: source (which conversation), type, confidence

**Scoring prompt**:
```
Extract information worth remembering long-term from the conversation below.
Score each (1-5): importance (impact on user's work/life)
  x persistence (will remain relevant) x reusability (needed again)
Return only items scoring >= 3, format: | score | type | content |
```

### Stage 2: REM (Reflector)

**Trigger**: Candidates accumulate >= 5 entries, or scheduled task

1. Fetch all pending candidates
2. For each candidate, retrieve similar historical memories from ChromaDB
3. LLM determines:
   - **Consistent with existing memory** → boost existing memory's importance score
   - **Contradicts existing memory** → flag conflict, queue for review
   - **Entirely new** → pass to deep sleep
   - **Low value / duplicate** → discard, update forgetting curve
4. Output REM report (auditable)

### Stage 3: Deep Sleep (Promoter)

**Trigger**: REM complete

1. Write validated candidates to:
   - MongoDB `memories` collection (long-term memory)
   - ChromaDB vector index (semantic retrieval)
2. Regenerate MEMORY.md (merge, used in system prompt)
3. Clean up processed candidates

### Forgetting Mechanism

```
Each memory has a decay field:
  - Initial value: 1.0
  - Daily decay: -0.02 (reaches 0 in ~50 days)
  - Retrieved/hit: +0.1 (cap 1.0)
  - Written to MEMORY.md: reset to 1.0
  - decay < 0.1: auto-cleanup on next consolidation
```

### Dream Configuration

```typescript
dream: {
  threshold: 5,           // candidates to trigger REM
  schedule: '0 3 * * *',  // cron schedule
  minScore: 3,            // minimum candidate score
  decay: { rate: 0.02, boost: 0.1, min: 0.1 },
  stages: { light: true, rem: true, deep: true }
}
```

## Learning Engine

### Experience Extraction

**Trigger**: `conversation.end` AND the conversation had >= 3 tool calls

1. Fetch full conversation record (user messages + tool call chain + final result)
2. LLM extracts structured experience:
   - Scenario description (what type of problem/task)
   - Solution method (which tools, what order, what parameters)
   - Key decisions (why this approach over alternatives)
   - Pitfalls encountered (what went wrong, how it was fixed)
   - Applicability conditions (when this method works)
3. Output as candidate Skill document (Markdown + YAML frontmatter)

### Skill Generation

1. Semantic comparison of candidate Skill with all existing skills (builtin + user-created + auto-generated)
2. Determine relationship:
   - **New** → create new skill, write to `cognitive_skills`
   - **Supplements existing** → append content to existing skill, version +1
   - **Replaces existing** → deprecate old version, create new
   - **Not worth summarizing** → discard
3. Based on `autoLevel` config:
   - `semi-auto`: `confirmedAt: null`, show in frontend for confirmation
   - `full-auto`: `confirmedAt: now()`, effective next turn

### Skill Iteration

**Trigger**: Existing skill is retrieved and used

1. Collect feedback after use: success / partial success / failure
2. Success → `confidence += 0.1`
3. Partial success / failure → record failure case, trigger LLM revision after 3 failures
4. `confidence < 0.3` → mark as low-confidence, downrank in retrieval

## Retrieval Engine

Replaces current `buildSystemPrompt()` full-stuff approach with on-demand retrieval.

```
query = user's latest message + current conversation summary (if any)
         │
         ├─→ BM25 search (minisearch, in-memory index)
         │     Index: MEMORY.md + last 30 days logs + active skills
         │     Output: top-20 by term frequency / relevance
         │
         ├─→ Vector search (ChromaDB)
         │     Index: long-term memory embeddings
         │     Output: top-20 by cosine similarity
         │
         └─→ Score fusion (RRF: Reciprocal Rank Fusion)
               score = sum(1 / (k + rank_i))  where k = 60
               → Time decay weighting: score * decay^days
               → Return top-K (default K = 8)
```

## System Integration

### Adapter Pattern (no changes to ConversationEngine core)

```typescript
// backend/src/cognitive/adapter.ts
export class CognitiveAdapter {
  // Replaces MemoryEngine.getMemoryContext()
  async getMemoryContext(userId: string): Promise<string> {
    return this.retrieval.query(userId, currentMessage);
  }

  // Replaces MemoryEngine.afterConversation()
  async afterConversation(userId: string, messages: ChatMessage[]) {
    this.bus.emit('conversation.end', { userId, messages });
    // Original logic (log append, commitment detection) preserved
  }

  // Replaces skill injection in buildSystemPrompt()
  async getSkillsContext(userId: string): Promise<string> {
    const skills = await this.skillEngine.getActiveSkills(userId);
    const cognitiveSkills = await this.learnEngine.getTopSkills(userId);
    return formatSkills([...skills, ...cognitiveSkills]);
  }
}
```

### Startup Sequence Change

```
Before: Config → MongoDB → MCP → Memory → Skill → Scheduler → Engine
After:  Config → MongoDB → Redis → ChromaDB → MCP → CognitiveCore → Engine
        CognitiveCore internally: Bus → Scorer → DreamEngine → LearnEngine → RetrievalEngine → Adapter
```

### Frontend Changes

1. **Memory panel enhanced**: show memory scores, decay status, dream history
2. **Skill panel enhanced**: new "Pending Skills" tab with auto-generated skills and confidence scores
3. **Settings panel**: cognitive configuration area (autoLevel, dream frequency, learning toggles)

## Phase 1 Scope (Minimum Viable Loop)

| Module | Phase 1 | Phase 2 |
|--------|---------|---------|
| MemoryScorer | Scoring + candidate write | — |
| DreamCollector (light sleep) | Candidate collection | — |
| DreamReflector (REM) | — | Cross-validation |
| DreamPromoter (deep sleep) | Direct promote (skip REM) | Validate-then-promote |
| LearnExtractor | Experience extraction | — |
| LearnGenerator | Skill generation (needs confirmation) | Auto-iteration |
| LearnIterater | — | Confidence tracking + revision |
| RetrievalEngine | BM25 only | + Vector hybrid |
| ChromaDB | Not introduced | Introduced |
| Redis | Candidate queue cache only | Full-layer caching |
| Forgetting | Simple TTL | Decay curve |

### Phase 1 Core Loop

```
Conversation end → extract candidate memories (score) → accumulate 5 → promote to long-term memory
Meanwhile: 3+ tool calls → extract experience → generate candidate skill → user confirms → available next turn
Retrieval: BM25 replaces full-stuff into prompt
```

## Error Handling

| Failure | Degradation |
|---------|-------------|
| LLM scoring call fails | Skip this turn, no candidate written; fall back to existing behavior |
| Redis unavailable | Write candidates directly to MongoDB; no working memory cache |
| ChromaDB unavailable (Phase 2) | Fall back to BM25-only retrieval |
| Scoring LLM returns malformed output | Skip this turn, log error, don't block conversation |
| BM25 index corruption | Rebuild from MongoDB; fall back to full-stuff prompt |
